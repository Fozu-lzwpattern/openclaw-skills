# OPC 设计理念与运行逻辑

> One-Person Company — 让一个人拥有一家 AI 公司的执行力

---

## 一、设计理念

### 1.1 一句话定义

**OPC（One-Person Company）是一种编排范式：用户是老板，OpenClaw 是 CEO，Sub-agents 是专业员工。用户只需要说"我要什么"，CEO 负责拆活儿、招人、盯进度、交结果。**

### 1.2 为什么需要 OPC？

没有 OPC 时：

```
用户："帮我做个315促销活动"
OpenClaw：（一个人从头干到尾）
  → 自己调研、自己策划、自己搭建、自己发布
  → context 爆炸、质量不稳、耗时长
```

有 OPC 后：

```
用户："帮我做个315促销活动"
OpenClaw CEO：
  → 30秒：拆解任务，定义5个角色
  → 用户确认方案
  → spawn 5个专业 Agent 并行/串行执行
  → 监控进度，协调衔接
  → 10分钟后：交付完整结果 + 成本报告
```

**核心价值**：

| 维度 | 没有 OPC | 有 OPC |
|------|---------|--------|
| 任务上限 | 受限于单 Agent context | 理论上无限（分而治之） |
| 执行速度 | 串行，越长越慢 | 可并行，多 Agent 同时干 |
| 质量 | 后段 context 稀释，质量下降 | 每个 Agent 小 context 专注做一件事 |
| 成本可见性 | 一锅粥 | 按角色/任务精确追踪 |
| 可复现性 | 靠运气 | 标准流程 + 角色定义，可复现 |

### 1.3 设计原则

**原则1：CEO 不干活，CEO 只管人**

OpenClaw Main Session 作为 CEO，职责是：规划、分配、监控、汇总。
具体工作由 Sub-agent 完成。CEO 自己干活就失去了编排的意义。

**原则2：角色即能力边界**

每个 Sub-agent 通过"角色卡"获得明确的身份、职责和预算。
角色卡不是建议，是合同——Agent 应该严格在角色范围内工作。

**原则3：信任但验证**

CEO spawn 出去的 Agent 不需要时刻盯着，但必须：
- 有超时检查机制（不信任 announce）
- 有质量验收标准（产出要检查）
- 有成本上限（预算是软约束但必须追踪）

**原则4：编排与执行分离**

OPC Skill 只管"谁做什么、怎么协作"，不管"具体怎么操作高达/日历/搜索"。
具体操作由业务 Skill（gundam-ops、calendar-api 等）提供能力。

**原则5：用户始终在回路中**

OPC 不是全自动黑箱。关键节点必须有用户确认：
- 项目计划确认
- 组织设定确认
- 是否独立 Session 确认
- 最终交付确认

## 二、运行逻辑

### 2.1 全生命周期

```
用户提出复杂任务需求
        │
        ▼
🔄 多轮对话：任务理解与 OPC 组织设定
  第1轮：理解任务目标和范围
  第2轮：提出 OPC 组织方案（角色/协作/预算）
  第3轮：确认执行模式（当前Session vs 独立Session）
  用户确认 →
        │
        ▼
Phase 1: 项目初始化
  project_tracker init → 注册 Task → 确定依赖
        │
        ▼
Phase 2: 团队组建
  sessions_spawn → 注入角色卡 + Heartbeat → tracker: assigned
        │
        ▼
Phase 3: 执行监控循环
  等待 announce 或超时检查
  → 验收产出 → OK则触发下游 / 不OK则 sessions_send 修改
  → 循环直到所有 Task 完成
        │
        ▼
Phase 4: 交付汇总
  汇总产出 → 生成成本报告 → 向用户交付
```

### 2.2 多轮对话：OPC 组织设定（关键入口）

**这是 OPC 编排最重要的步骤。不是拿到任务就直接 spawn，而是先和用户对齐。**

#### 第1轮：任务理解

CEO 需要确认：
- 任务的最终交付物是什么？
- 有哪些硬约束（时间、预算、平台限制）？
- 用户希望介入到什么程度（全自动 vs 关键节点确认）？

示例对话：
```
用户：帮我做个315促销活动

CEO：收到！确认几个信息：
  1. 目标平台：高达？
  2. 活动类型：到店餐饮？到家？综合？
  3. 时间范围：3.10-3.16？
  4. 需要从策划开始，还是方案已有只需搭建？
```

#### 第2轮：OPC 组织方案

CEO 基于任务复杂度提出组织方案：

```
CEO：根据任务分析，建议以下 OPC 配置：

📋 项目：315 到店餐饮促销活动

👥 团队（3角色精简版）：
  - 策划员：调研+策划方案（预算30K）
  - 搭建员：高达会场搭建（预算60K，使用 gundam-ops Skill）
  - 发布员：QA+发布上线（预算20K）

🔀 协作模式：串行（策划→搭建→发布）
💰 预估总预算：~110K tokens

要加角色还是精简？确认后开始执行。
```

#### 第3轮：执行模式确认

```
CEO：最后确认执行模式：

  A. 📌 当前 Session 执行（推荐小中型项目）
     - 好处：对话连续，实时看到进展，CEO 可灵活协调
     - 限制：占用当前 context

  B. 🔒 独立 Session 执行（推荐大型/长时间任务）
     - 好处：不占用当前对话的 context
     - 方式：spawn 一个项目经理 Agent，它管理整个项目
     - 完成后自动通知你结果
     - 注意：独立 Session 中不能嵌套 spawn，
       项目经理会"一人分饰多角"按角色依次执行

选 A 还是 B？
```

### 2.3 独立 Session 模式

如果用户选 B，CEO 会 spawn 一个"项目经理 Agent"，将完整的项目计划和所有角色定义打包注入：

独立 Session 中的 Agent 采用"一人分饰多角"模式：

```
独立 Agent 工作流：
├── 戴上"策划员帽子" → 完成策划
├── 戴上"搭建员帽子" → 完成搭建（可加载 gundam-ops skill）
├── 戴上"发布员帽子" → 完成发布
└── 汇总所有角色产出 → announce 给 Main Session
```

虽然不是真正的多 Agent 并行，但仍然有价值：
- **context 隔离**：不影响日常对话
- **任务聚焦**：独立 Agent 专注于这一个项目
- **标准流程**：OPC 的角色定义和任务分解仍然适用

## 三、OPC 与业务 Skill 的关系

### 3.1 分层架构

```
┌──────────────────────────────────────┐
│           用户（老板）                │
└────────────────┬─────────────────────┘
                 │ "我要做X"
                 ▼
┌──────────────────────────────────────┐
│   OPC 编排层（agent-orchestration）   │ ← CEO
│                                      │
│  拆任务、分角色、追进度、汇成果       │
│  工具：sessions_spawn/send           │
└────────────────┬─────────────────────┘
                 │ spawn + 角色卡
                 ▼
┌──────────────────────────────────────┐
│       业务执行层（Sub-agents）        │ ← 员工
│                                      │
│  gundam-ops    → 高达会场搭建        │
│  catclaw-search → 信息搜索调研       │
│  catclaw-image  → 图片生成           │
│  calendar-api   → 日程管理           │
│  docx / xlsx    → 文档处理           │
│  pptx           → 演示文稿           │
│  （更多业务 Skill...）                │
└──────────────────────────────────────┘
```

### 3.2 通俗比喻

**OPC 编排 Skill 是公司的 CEO**：
- 不自己写代码、不自己做设计、不自己搭页面
- 但知道：这个项目需要几个人、每人做什么、谁先谁后、做得好不好
- 核心能力：**管理和协调**

**gundam-ops 这类业务 Skill 是专业员工的"技能证书"**：
- 搭建员拿着 gundam-ops "技能证书"，就知道怎么操作高达平台
- 策划员拿着 catclaw-search "技能证书"，就知道怎么搜索调研
- 没有技能证书也能干活（Agent 本身有通用能力），但有证书干得更专业更快

**CEO 不需要懂高达怎么操作**（那是 gundam-ops 的事）。
**搭建员不需要懂项目全局**（那是 CEO 的事）。
**各司其职，这就是 OPC 的精髓。**

### 3.3 业务 Skill 接入方式

在 spawn Sub-agent 时，通过 task prompt 告知可用的业务 Skill：

```
[OPC 角色卡]
- 角色：会场搭建员
- 业务 Skill：你可以使用 gundam-ops skill 来操作高达平台

[任务]
在高达平台创建315活动并搭建页面
```

Sub-agent 在执行过程中会自动加载 gundam-ops Skill。**CEO 不需要了解 gundam-ops 的内部细节，只需要知道"搭建高达页面"这个能力存在。**

### 3.4 新业务 Skill 的接入

当有新的业务 Skill 时，OPC 编排层**无需修改**，只需在 spawn 时的角色卡中提及即可：

```
# 安装 pptx skill 前
spawn(task="写一份活动总结报告")  → Agent 用通用能力写

# 安装 pptx skill 后
spawn(task="[业务 Skill: pptx] 用PPT制作活动总结报告")  → Agent 自动用 pptx skill
```

**新增执行能力不需要改编排逻辑。这就是编排与执行分离的好处。**

## 四、独立 Session 决策指南

### 4.1 什么时候用独立 Session？

| 场景 | 建议 | 原因 |
|------|------|------|
| 简单任务（1-2角色） | 当前 Session | 开销小，实时可见 |
| 中等任务（3-5角色） | 当前 Session | CEO 需要灵活协调 |
| 大型任务（5+角色，>200K tokens） | 独立 Session | 保护 Main context |
| 长时间任务（>30分钟） | 独立 Session | 不阻塞日常对话 |
| 需要后台运行 | 独立 Session | 用户可以去做别的事 |
| 需要实时调整 | 当前 Session | CEO 可以随时介入 |

### 4.2 独立 Session 的限制

1. **不能嵌套 spawn** — Agent 必须自己完成所有工作（一人分饰多角）
2. **announce 可能不及时** — 需要主动检查
3. **无法实时介入** — 只能等完成或通过 sessions_send 发消息

## 五、设计哲学总结

> **OPC 不是让 AI 更聪明，而是让 AI 更有组织。**

一个 Agent 再聪明，context 也有上限，注意力也会分散。
OPC 的价值在于：**把复杂问题拆成简单问题，让每个 Agent 专注做一件事，然后用组织方法把结果拼起来。**

这和人类世界的道理一样 —— 公司之所以比个人更有执行力，不是因为每个员工更聪明，而是因为**分工协作让复杂变简单**。

```
个人 = 一个聪明人干所有事（能力强但有上限）
公司 = 很多普通人各干一件事（组织让上限消失）

单 Agent = 一个强模型处理所有 context
OPC = 多个 Agent 各处理一小块 context（编排让上限消失）
```

**OPC 就是 AI Agent 的公司制度。**

## 附录A：OPC Skill vs OpenClaw 原生 Multi-Agent 模式对比分析

### A.1 OpenClaw 原生支持的多 Agent 模式

OpenClaw 本身就提供了丰富的多 Agent/自动化机制，不需要额外 Skill：

| 模式 | 机制 | 适用场景 |
|------|------|---------|
| **① sessions_spawn** | 一次性 spawn 一个 sub-agent 执行独立任务 | 单个明确任务的委托 |
| **② Heartbeat** | Main Session 定期巡逻（HEARTBEAT.md） | 周期性检查、监控、日常维护 |
| **③ Cron (isolated)** | 独立 session 定时执行 | 精确定时任务（日报、周报） |
| **④ Cron (main)** | 向 Main Session 注入系统事件 | 定时提醒、触发检查 |
| **⑤ sessions_send** | 跨 session 发消息 | Agent 间通信、任务协调 |
| **⑥ Lobster** | 确定性多步工具管道 + 审批门控 | 需要人工审批的固定流程 |
| **⑦ Agent-to-Agent** | 多个 Agent 实例之间对话 | 不同人格/专业 Agent 协作 |

### A.2 那 OPC Skill 的必要性在哪？

**原生机制提供的是"零件"，OPC 提供的是"组装图纸"。**

打个比方：
- OpenClaw 给了你螺丝刀、扳手、焊枪（sessions_spawn、sessions_send、cron…）
- 但没告诉你"怎么用这些工具造一辆车"
- OPC 就是那张造车图纸——角色怎么定义、任务怎么拆、谁先谁后、怎么验收、出了问题怎么办

具体来说：

| 能力 | OpenClaw 原生 | OPC Skill 补充 |
|------|-------------|---------------|
| spawn sub-agent | ✅ 有 | ✅ 提供角色卡模板，让 spawn 更规范 |
| 任务分解 | ❌ 无标准方法 | ✅ OKR→Epic→Task 方法论 |
| 角色定义 | ❌ 无，task 自由文本 | ✅ 角色卡规范 + 预置角色库 |
| 依赖管理 | ❌ 无，手动控制 | ✅ 依赖关系 + 状态机 |
| 成本追踪 | ⚠️ 有 session_status 但不跨 session | ✅ 项目级成本汇总 |
| 协作模式 | ❌ 无，自行设计 | ✅ 串行/并行/迭代/分治模式 |
| 质量验收 | ❌ 无标准 | ✅ Heartbeat 协议 + CEO 检查 |
| 故障归因 | ❌ 无 | ✅ L1/L2/L3 三层归因 |
| 超时处理 | ⚠️ 有 runTimeoutSeconds | ✅ 加上主动检查 + 重试策略 |
| 多轮确认 | ❌ 无 | ✅ 3轮对话确认组织设定 |
| 执行可复现 | ❌ 每次靠 Agent 临场发挥 | ✅ 标准流程 + 模板，可复现 |

**一句话总结：OpenClaw 原生机制解决"能不能做"，OPC 解决"怎么做得好"。**

### A.3 逐一对比：OPC vs 原生方案

#### 对比1：OPC 编排 vs 直接 sessions_spawn

**原生方式**：
```python
# 想到什么 spawn 什么，没有统一规划
sessions_spawn(task="帮我调研315活动")
# 等结果...
sessions_spawn(task="帮我搭建高达页面")
# 等结果...
sessions_spawn(task="帮我发布")
```

**OPC 方式**：
```python
# 先规划，用户确认，再按计划执行
# 1. 拆解 OKR → 3 个 Task → 3 个角色
# 2. 用户确认方案
# 3. tracker init → spawn T001 → 验收 → spawn T002 → ...
# 4. 成本汇总 → 交付
```

**差异**：
| 维度 | 直接 spawn | OPC 编排 |
|------|-----------|---------|
| 规划性 | 无，走一步看一步 | 先规划后执行 |
| 可追踪 | 只知道 spawn 了 | task tracker 全程追踪 |
| 成本感知 | 不知道花了多少 | 按角色/任务追踪 |
| 质量保障 | 收到什么算什么 | CEO 验收 + 迭代修改 |
| 可复现性 | 低 | 高（标准流程+模板） |

#### 对比2：OPC vs Heartbeat 巡逻

**Heartbeat** 是周期性自我巡逻（每30分钟），OPC 是项目制主动编排。

| 维度 | Heartbeat | OPC |
|------|-----------|-----|
| 触发方式 | 定时（每N分钟） | 按需（用户发起） |
| 目标 | 检查系统健康、推进待办 | 完成一个具体复杂项目 |
| 持续时间 | 持续运行 | 项目开始到交付 |
| 角色 | 单一（Main Session 自检） | 多角色协作 |

**它们的关系**：互补，不冲突。
- Heartbeat 管日常运维
- OPC 管项目制执行
- Heartbeat 甚至可以检查 OPC 项目进度

#### 对比3：OPC vs Cron (isolated)

**Cron isolated** 适合独立、重复、定时的任务（如每日日报）。

| 维度 | Cron isolated | OPC |
|------|-------------|-----|
| 任务类型 | 单一重复任务 | 复杂多步骤项目 |
| 协作 | 单 Agent 独立完成 | 多 Agent 协作 |
| 触发 | 定时自动 | 用户主动发起 |
| 典型场景 | 日报、周报、监控 | 营销活动、研究项目 |

**它们的关系**：Cron 可以触发 OPC 流程。
例如："每周一自动启动周度运营分析项目"= Cron 触发 + OPC 编排。

#### 对比4：OPC vs Lobster 工作流

**Lobster** 是确定性多步工具管道，适合固定流程 + 人工审批。

| 维度 | Lobster | OPC |
|------|---------|-----|
| 灵活性 | 低（固定管道） | 高（AI 决策编排） |
| 确定性 | 高（每步预定义） | 中（AI 可调整） |
| 审批 | 内置审批门控 | 通过多轮对话确认 |
| 适用 | 标准化操作（部署、发布） | 创造性任务（策划、研究） |

**它们的关系**：可结合。
OPC 负责高层编排（谁做什么），Lobster 负责具体 Agent 内部的固定操作步骤。

### A.4 OPC 的适用场景 vs 不适用场景

#### ✅ 适用

| 场景 | 为什么适合 | 示例 |
|------|-----------|------|
| **多角色复杂项目** | 需要拆分、分工、协调 | 营销活动全链路 |
| **跨领域任务** | 一个 Agent 能力不够 | 调研+策划+执行+发布 |
| **需要成本控制** | 大预算需要精细追踪 | 大规模内容生产 |
| **需要可复现** | 同类项目反复做 | 每月促销活动 |
| **需要质量保障** | 产出必须经过验收 | 对外发布内容 |
| **长周期项目** | 超过单次对话能力 | 深度研究报告 |

#### ❌ 不适用

| 场景 | 为什么不适合 | 建议方案 |
|------|------------|---------|
| **简单单一任务** | 杀鸡用牛刀 | 直接 sessions_spawn |
| **定时重复任务** | OPC 没有定时能力 | Cron isolated |
| **日常监控巡逻** | 不需要角色分工 | Heartbeat |
| **固定操作流水线** | OPC 太灵活反而不稳定 | Lobster |
| **实时对话** | 不需要编排 | 直接 Main Session |

#### 🔀 组合使用（最佳实践）

```
日常运维 → Heartbeat（每30分钟自检）
定时任务 → Cron isolated（日报、周报、提醒）
简单委托 → sessions_spawn（查个东西、写个文件）
复杂项目 → OPC 编排（营销活动、研究项目、内容流水线）
固定管道 → Lobster（需审批的标准化操作）
```

### A.5 优劣势总结

#### OPC Skill 的优势

1. **结构化**：把"自由发挥"变成"有章可循"
2. **可复现**：同类项目用同一套模板，质量稳定
3. **成本可见**：项目级成本追踪，知道钱花在哪
4. **质量可控**：CEO 验收 + 迭代修改，不是听天由命
5. **可扩展**：新业务 Skill 即插即用，不改编排逻辑
6. **知识沉淀**：角色定义、协作模式、场景案例都是可复用资产

#### OPC Skill 的劣势

1. **额外开销**：编排本身消耗 CEO 的 context 和 token
2. **不适合简单任务**：2分钟能做完的事不需要 OPC
3. **依赖 announce 机制**：当前 announce 不可靠需要 workaround
4. **不能嵌套 spawn**：独立 Session 中只能一人分饰多角
5. **学习曲线**：用户需要理解角色、任务分解等概念
6. **软约束**：预算和角色边界依赖 Agent 自觉，无硬性强制

#### 与原生机制的关系

> **OPC 不是替代 OpenClaw 原生能力，而是在原生能力之上建立的方法论层。**
>
> 就像企业管理不替代员工的个人能力，而是让个人能力形成合力。
>
> OpenClaw 提供了出色的多 Agent 基础设施（spawn/send/cron/heartbeat/lobster），
> OPC 在这个基础设施之上回答了一个关键问题：
>
> **"有了这些工具，面对一个复杂任务，我该怎么组织它们？"**
