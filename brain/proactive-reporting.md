# CEO 主动监控与汇报机制

## 问题背景

联调中反复出现的问题：Sub-agent 完成任务后 announce 不到达 CEO，
导致项目停滞等待——直到用户主动询问才发现 Agent 早已完成。

这违背了 OPC "用户不应该需要催" 的核心原则。

## 解决方案：CEO 主动轮询机制

### 机制设计

```
CEO spawn sub-agent 后，立即启动监控循环：

1. 记录 spawn 时间和预估完成时间
2. 设置检查间隔（根据任务复杂度）
3. 到达检查点时，主动执行：
   a. subagents(action=list) 查看状态
   b. 如果 done → sessions_history 拉取结果 → 推进下游
   c. 如果 active → 检查运行时长是否超预期
   d. 如果 error → 分析原因，决定重试/告警
4. 向用户推送进度更新
```

### 检查间隔标准

| 任务复杂度 | 预估时长 | 首次检查 | 后续间隔 |
|-----------|---------|---------|---------|
| 简单（搜索/整理） | 2-5min | 3min 后 | 每 2min |
| 中等（策划/分析） | 5-15min | 5min 后 | 每 3min |
| 复杂（搭建/多步骤） | 15-30min | 10min 后 | 每 5min |

### CEO 主动汇报规则

**规则1：spawn 后必须给用户一条确认**
```
"✅ 策划员已启动（预计5分钟），我会持续监控并汇报进展。"
```

**规则2：到达检查点必须汇报**
```
- 如果 Agent 还在跑："🔄 策划员运行中（已3分钟），正在搜索调研阶段。"
- 如果 Agent 已完成："✅ 策划员完成！产出质量OK，正在启动搭建员。"
- 如果 Agent 异常："⚠️ 策划员运行超时（15分钟），正在检查..."
```

**规则3：阶段转换必须汇报**
```
"📊 项目进度更新：
 T001 策划 ✅ → T002 制券 🔄 → T003 选品 🔄 → T004 搭建 ⏳ → T005 发布 ⏳"
```

**规则4：异常必须立即汇报**
```
"⚠️ 搭建员执行失败（错误：SSO 认证过期），需要你帮忙处理。"
```

### 实现方式

由于 OpenClaw 中 CEO（Main Session）不能设置定时器，
主动监控需要 CEO 在每次与用户交互时顺带检查：

```
用户发任何消息时：
1. 先检查有没有正在运行的 OPC 项目
2. 如果有 → subagents list → 更新状态 → 必要时汇报
3. 再处理用户当前消息
```

或者通过 heartbeat 机制：

```
Heartbeat 巡逻时：
1. 检查有没有 active sub-agents
2. 如果有 → 检查运行时长 → 必要时拉取结果或告警
```

### 汇报模板

**进度更新（定期）**
```
📊 {项目名} 进度更新

| Task | 状态 | 时长 | Token |
|------|------|------|-------|
| T001 | ✅ done | 3min | 10K |
| T002 | 🔄 running | 2min | - |
| T003 | ⏳ pending | - | - |

下一步：T002 预计 2 分钟完成，届时启动 T003。
```

**异常告警**
```
⚠️ {项目名} 异常

问题：{Agent角色} 运行 {时长}，超过预期 {预期时长}。
状态：{subagents list 结果}
建议：{重试 / kill 重建 / 等待 / 需要用户介入}
```

**阶段完成通知**
```
✅ {项目名} 阶段完成

{Agent角色} 已完成 {任务名}：
- 产出：{摘要}
- 消耗：{token}K tokens
- 质量：{OK / 需修改}

下一步：正在启动 {下游任务}。
```

## 与 Heartbeat 协议的关系

Sub-agent 端的 Heartbeat 协议（汇报产出）+ CEO 端的主动监控机制 = 双保险。

- **Sub-agent Heartbeat**：Agent 完成后主动汇报（依赖 announce，不可靠）
- **CEO 主动监控**：CEO 定期检查 Agent 状态（不依赖 announce，可靠）

两者互补，确保不出现"两边都在等对方"的死锁。
